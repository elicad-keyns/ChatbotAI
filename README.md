# Chatbot AI Desktop

Desktop-приложение на Tauri + React с отдельной сущностью Agent, которая принимает запрос пользователя, вызывает OpenAI API и возвращает ответ в интерфейс.

## Что уже есть

- React UI с чатом и панелью настроек.
- Tauri command `send_agent_message` со streaming-ответом через события `agent_stream_delta`.
- Rust-модуль `agent`, инкапсулирующий логику запроса к LLM.
- Поддержка OpenAI API key через поле в UI или переменную окружения `OPENAI_API_KEY`.

## Как запустить

1. Установите Node.js LTS.
2. Установите Rust: https://www.rust-lang.org/tools/install
3. Установите зависимости:

```bash
npm install
```

4. Запустите desktop-приложение:

```bash
npm run tauri:dev
```

Можно не вводить ключ в UI, если перед запуском задана переменная окружения `OPENAI_API_KEY`.

## Где находится Agent

Основная логика агента находится в:

```text
src-tauri/src/agent.rs
```

Frontend вызывает агента через Tauri command:

```text
src-tauri/src/lib.rs
```

## День 11. Модель памяти ассистента

В проект добавлена явная модель памяти с тремя слоями:

- Краткосрочная память: сообщения текущего чата. Хранится внутри выбранного `ChatSession` в `chatbot-ai.chats.v1` и меняется при переключении чатов.
- Рабочая память: временный контекст активной задачи во всех чатах: текущая фича, баг, файлы, ограничения, решения на ближайшую работу. Хранится отдельно в `chatbot-ai.memory.working.v1`.
- Долгосрочная память: стабильный профиль пользователя во всех чатах: профессия, навыки, языки общения и программирования, цели, проекты, предпочтения и характеристики, влияющие на работу. Хранится отдельно в `chatbot-ai.memory.long-term.v1`.

Сохранение в память делается автоматически через OpenAI memory-router:

- любое отправленное сообщение всегда попадает в краткосрочную память текущего чата;
- краткосрочную память можно сжимать: в настройках включается компрессия и лимит свежих ходов без сжатия, где один ход = запрос пользователя + ответ ИИ;
- если лимит превышен, backend создает или обновляет `shortTermSummary`: берет предыдущее summary и следующий старый завершенный ход `user + assistant`, затем отправляет в основной OpenAI-запрос summary плюс свежий хвост сообщений;
- полная визуальная история чата не удаляется и остается в `ChatSession`, сжимается только контекст, который уходит в OpenAI-запрос;
- после основного ответа backend делает короткий дополнительный OpenAI-вызов на `gpt-4.1-mini`;
- роутер получает последнее сообщение пользователя, сгенерированный ответ ассистента, текущую рабочую память и текущую долгосрочную память в компактном TOON-like виде;
- роутер возвращает несколько компактных строк TOON-like формата, по одной на независимый факт или рабочий артефакт: `working|save|...|...`, `longTerm|save|...|...`;
- для рабочей памяти нет жесткого лимита в 6 строк: LLM сама выбирает количество записей и может использовать до 1000 output-токенов, сохраняя только критичный контекст, без которого потеряется часть задачи;
- если LLM решила `save` для рабочей памяти, frontend сохраняет краткую выжимку состояния активной задачи после ответа ассистента: цель, решения, артефакт, экраны, функции, требования, стек или ограничения;
- если LLM решила `save` для долгосрочной памяти, frontend сохраняет краткую выжимку как профиль, навык, язык, цель, проект, предпочтение, личную характеристику или устойчивый факт пользователя;
- если сообщение смешанное, LLM должна разделить факты: личное в `longTerm`, проектный результат в `working`;
- несвязанные долгосрочные факты сохраняются отдельными записями;
- похожие записи не дублируются;
- удалить ошибочную запись можно вручную из правой панели.

Backend получает `memoryContext` вместе с сообщениями. Rust-агент добавляет к `instructions` отдельный блок `Explicit assistant memory model`, где описаны границы слоев, краткосрочное summary, рабочая и долгосрочная память. Краткосрочная память передается как свежий хвост сообщений текущего чата, а старая часть при включенной компрессии передается как summary.

Основной ответ ассистента идет через stream: backend читает SSE-события OpenAI Responses API и отправляет текстовые дельты во frontend через Tauri event `agent_stream_delta`. После завершения текста backend отправляет событие `agent_memory_started`, запускает OpenAI memory-router и frontend показывает loader создания памяти.

Для проверки работы добавлена debug-панель:

- показывает счетчики `shortTerm`, `working`, `longTerm`;
- показывает последние решения memory-router: сохранено, пропущено или удалено;
- показывает debug компрессии краткосрочной памяти: полный размер истории, размер фактического input, число сжатых ходов, input и raw-output summary-вызова;
- после ответа агента показывает, какие memory layers вошли в запрос;
- показывает фрагмент memory-инструкции, которая ушла в backend.
- показывает input и raw-output memory-router, чтобы было видно, на основе чего LLM решила сохранять память.

Окно приложения ограничено высотой viewport. История чатов, сообщения и правая панель памяти/отладки скроллятся внутри своих областей, поэтому весь документ не должен расползаться по высоте.

Проверка сборки:

```bash
npm run build
cd src-tauri
cargo check
```

## День 16. Подключение MCP

В приложении реализован MCP-клиент для локальных `stdio`-серверов и удалённых
`Streamable HTTP`-серверов. Клиент:

1. запускает процесс из настроек (`command`, `args`, `env`, `cwd`);
2. отправляет JSON-RPC запрос `initialize`;
3. отправляет уведомление `notifications/initialized`;
4. вызывает `tools/list` с поддержкой пагинации;
5. показывает полученные инструменты в настройках и передает их агенту;
6. вызывает `tools/call`, если пользователь отправляет прямую MCP-команду.

Для Streamable HTTP клиент отправляет JSON-RPC через POST, принимает JSON или SSE,
сохраняет `Mcp-Session-Id`, передаёт `MCP-Protocol-Version` и поддерживает
пользовательские HTTP-заголовки (`Authorization`, API keys и другие).

Локальный тестовый сервер `@modelcontextprotocol/server-everything` установлен как
dev-зависимость и уже добавлен в настройки по умолчанию.

### Проверка в приложении

```bash
npm install
npm run tauri:dev
```

Откройте `Настройки` → `MCP-серверы` и нажмите
`Проверить и получить tools`. При успешной проверке интерфейс покажет статус
соединения и имена инструментов, возвращенных `tools/list`.

Кнопки `+ Remote` и `+ stdio` создают конфигурации соответствующего транспорта.
Для каждого stdio-сервера
можно отдельно задать название, команду, аргументы (по одному на строку), рабочую
папку и переменные окружения. Для remote-сервера задаются endpoint URL и
HTTP-заголовки. Включенных серверов может быть несколько.

Публичный пример удалённого сервера:

```text
Name: Cloudflare Docs
Transport: Streamable HTTP
URL: https://docs.mcp.cloudflare.com/mcp
```

Для Figma доступен preset `+ Figma` с URL `https://mcp.figma.com/mcp`. Figma Remote
требует интерактивный OAuth и на текущем этапе принимает только MCP-клиенты из
каталога Figma. Generic HTTP-транспорт реализован, но для полноценной авторизации
собственного приложения потребуется зарегистрировать его у Figma.

Прямой вызов инструмента из чата:

```text
/mcp <serverId> <toolName> {"argument":"value"}
```

Для инструмента с уникальным именем доступна короткая форма:

```text
/mcp <toolName> {"argument":"value"}
```

Пример для сервера Everything:

```text
/mcp everything echo {"message":"hello"}
```

### Автоматическая проверка реального соединения

После `npm install` можно запустить интеграционный тест. Он поднимает локальный
Everything MCP, устанавливает соединение и печатает список инструментов:

```bash
cd src-tauri
cargo test mcp::tests::everything_server_connects_and_lists_tools -- --ignored --nocapture
cargo test mcp::tests::remote_http_server_connects_and_lists_tools -- --ignored --nocapture
```

Основная реализация находится в `src-tauri/src/mcp.rs`, Tauri-команда проверки —
в `src-tauri/src/lib.rs`, интерфейс нескольких подключений — в `src/App.tsx`.

## День 17. Yandex Tracker MCP

Собственный MCP-сервер создан как независимый Python-проект в соседней папке:

```text
../YandexTrackerMCP
```

Он может быть вынесен в отдельный репозиторий и развёрнут на сервере через Docker.
Сервер регистрирует инструменты `create_issue`, `get_issue`, `update_issue`,
`list_issue_transitions` и `cancel_issue`, описывает входные параметры через JSON Schema
и возвращает структурированный результат с ключом и ссылкой задачи.

В настройках приложения кнопка `+ Tracker` добавляет готовую remote-конфигурацию.
После деплоя замените локальный URL на `https://your-host/mcp`, укажите заголовок
`Authorization: Bearer <MCP_API_KEY>`, включите сервер и выполните проверку tools.

Агент автоматически выбирает подходящий MCP-инструмент через function calling. Для
write-инструментов он сначала запрашивает подтверждение, а после ответа пользователя
`да` восстанавливает параметры из истории, выполняет `tools/call` и сообщает об успехе
только при наличии фактического результата с ключом и URL задачи.

Прямой проверочный вызов:

```text
/mcp yandex-tracker create_issue {"summary":"Day 17 MCP demo","queue":"TEST","confirmed":true,"unique":"day17-tracker-demo"}
```

Инструкция OAuth/IAM, Docker и production-секретов находится в
`../YandexTrackerMCP/README.md`. Отдельные задачи в Tracker удалить нельзя, поэтому
инструмент `cancel_issue` выполняет подтверждённый workflow-переход отмены или закрытия.

## День 18. Планировщик и Telegram

Тот же `YandexTrackerMCP` теперь регистрирует инструменты расписаний и отчётов. Агент
может естественным языком создать cron/interval/once задание, поставить его на паузу,
запустить вручную и получить сохранённую сводку. После подтверждения агент вызывает
MCP-инструмент автоматически.

Расписания, история запусков и агрегированные отчёты сохраняются в SQLite внутри
`../YandexTrackerMCP/data/scheduler.db`. Отдельный сервис
`../YandexTrackerTelegramBot` принимает готовые отчёты, хранит подписчиков и копии
доставок в своей SQLite и поддерживает команды `/tasks`, `/task`, `/report`, `/history`
и `/jobs`.
